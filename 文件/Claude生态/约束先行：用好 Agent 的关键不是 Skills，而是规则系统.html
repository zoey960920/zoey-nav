用好Agent最重要的技巧不是Skills，是这四个字。

作者：数字生命卡兹克 · 发布时间：2026-04-14 10:09 CST

一句话结论
没有写进文档，等于没有真正交给 Agent。

作者用“公司治理”和“城市路网”做类比，强调规则是主干道，执行只是车辆在路上跑。

问题从哪里暴露出来

作者不是先谈理论，而是先从一个混乱的工作区翻车案例讲起。

触发点

  • 一个专门给 Claude Code 开发 Skills 的文件夹，几天后就变得根目录散乱。
  • 打包文件、源码、测试图片、评估 HTML 混在一起，连作者自己都认不出产物归属。
  • 命名开始失真，`test_batch`、`test_v2` 这类名字失去语义，工作空间无法复用。
  • 真正的问题不是“懒得整理”，而是 Agent 进入新空间时默认直接开工，没有先立规则。

文件混乱

目录失序，后续查找成本指数上升。

命名失真

产物缺少语义，过两天连自己都看不懂。

上下文丢失

对话一关，Agent 唯一能依赖的只剩文档与记忆。

核心判断

为什么作者说“最重要的不是 Skills”？因为 Skills 解决的是能力，规则解决的是秩序。

结论翻译成业务话

Prompt 像一次会议上的口头安排，Skill 像部门工具箱，约束体系才像公司的制度与 SOP。制度不到位，工具越多，混乱只会放大。

四层规则体系

这篇文章给出的不是一句口号，而是一套自上而下穿透的结构。

L1 / 全局规则

先定义“你是谁、怎么协作、什么算好”。

这是最高指令层。它不应该堆具体操作，而应该写长期稳定的原则，例如默认中文、先结论后解释、要解释为什么、用户体验优先于技术偏好。

这一层应该写什么

写价值观、判断框架、协作口径、输出标准。不要把项目路径、专题命名、某次活动流程塞进这里，否则全局规则会快速臃肿。

AI 使用原则

作者真正给出的，不是 100 条技巧，而是 6 条足够影响执行质量的原则。

1. 新任务先建规则，再开始干活。

任何项目，不论是开发、研究、画图还是做 Skills，都不默认直接开做。

2. 规则要覆盖非开发工作。

知识管理、报告制作、素材整理，一样需要目录、命名和清理约束。

3. 顶层规则要瘦身。

只放长期有效的原则，把细节下沉到项目层和过程文档层。

4. 改规则先改文档，再改行为。

不要让 Agent 先绕过规范赶进度，事后再补说明。

5. 要求 Agent 解释“为什么”和“用户影响”。

不只讲实现方案，还要告诉你为什么这么做、用户会感受到什么。

6. 把 UX 写进规则，不靠现场发挥。

复杂性由系统承担，反馈必须告诉用户下一步，而不是只报错。

今天就能执行

把这篇文章翻译成 Wayne 体系里可落地的动作，不需要大工程，先做 3 级固化就够。

全局协作口径

把“中文、先结论、要解释为什么、不要谄媚、交付时说风险边界”固定在顶层。

用户体验红线

把“复杂性由系统承担”“反馈要导向下一步”写进长期规则，而不是每次重复提醒。

风格与发布偏好

像 `wwei.ai` 发布、报告默认单 HTML、引用要带来源，这些都属于高频偏好，应被固化。

项目级 AGENTS

只写项目目标、执行步骤、交付方式、发布与归档动作，不复制全局规则。

模板先行

研究报告、网页演示稿、市场分析页都应有起步模板，让 Agent 从“套轨道”开始，而不是临时拼凑。

记忆下沉

阶段性经验写进文档或记忆层，避免知识只留在对话气泡里。

统一命名与归档

项目目录、交付物命名、素材归档口径统一，减少“看得见产出、看不见秩序”的隐性损耗。

SOP 与复盘闭环

每次翻车都不止修结果，还要反写规则，这样 Agent 和团队都会越用越稳。

把制度当产品做

制度本身也要考虑体验，写得能执行、能复用、能被新人快速理解。

记忆点
Prompt 像方向盘，规则像城市路网。方向盘再灵，路网歪了也只会越开越乱。

这也是这篇文章最有传播力的地方。它不是在教你“如何更会用 AI”，而是在提醒你：任何复杂系统的效率，最后都取决于结构和秩序。

对个人

减少重复提醒，让高频偏好真正被继承。

对项目

让 Agent 进入任何目录时先看宪法，再动手执行。

对团队

把隐性经验变成显性制度，减少返工和口口相传的损耗。

来源与收束

这篇文章最值得保留的，不是措辞，而是它对 Agent 协作本质的重新定义。

四条最终结论

1. Agent 的上限，不只取决于模型能力，更取决于规则体系是否成型。
2. 规则是跨会话资产，Prompt 只是即时操作。
3. 治理 Agent，本质上和治理团队、治理项目一样，都是先搭结构再扩执行。
4. “约束先行”不是保守，而是为了更稳定、更高速、更少返工地创造。
